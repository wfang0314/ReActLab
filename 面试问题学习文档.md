## 1. 项目整体架构

### 1.1 一句话概括
这是一个**智能问答+故障诊断**的机器人系统，用户可以问问题，也可以让它自动诊断系统故障。

## 2. RAG和AIOps两个模块的区别

### 2.1 RAG问答（Retrieval-Augmented Generation）

#### 工作流程（简单版）
```
用户提问 → 找相关文档 → 大模型阅读 → 生成答案
```

#### 代码位置
`app/services/rag_agent_service.py`

#### 详细流程
```python
# 第1步：用户问问题
question = "Redis连接超时怎么办？"

# 第2步：去Milvus找相关的文档片段（RAG的R）
related_docs = search_milvus(question)  # 返回最相似的3段文字

# 第3步：把文档和问题一起给大模型
prompt = f"根据以下资料回答问题：\n{related_docs}\n\n问题：{question}"

# 第4步：大模型生成答案（RAG的G）
answer = chat_qwen.generate(prompt)
```

#### 特点
- **单次交互**：问一次答一次
- **被动响应**：用户问才回答
- **依赖文档**：答案基于知识库内容

---

### 2.2 AIOps诊断

#### 工作流程（简单版）
```
发现告警 → 制定计划 → 执行检查 → 分析结果 → 生成报告
     ↑______________________________________________↓
                    （可能需要多轮）
```

#### 代码位置
`app/services/aiops_service.py`

#### 详细流程
```python
# 第1步：发现系统有告警（如CPU过高）

# 第2步：Planner制定诊断计划
plan = [
    "查询近1小时的错误日志",
    "获取当前CPU和内存使用率",
    "检查数据库连接池状态",
    "分析告警模式"
]

# 第3步：Executor逐个执行
for step in plan:
    result = execute(step)  # 调用MCP工具执行
    
    # 第4步：Replanner评估结果
    if need_more_info():
        update_plan()  # 调整计划，继续执行
    else:
        break

# 第5步：生成诊断报告
report = generate_report(all_results)
```

#### 特点
- **多步骤流程**：不是一步完成
- **主动推理**：Agent自己决定下一步做什么
- **动态调整**：根据中间结果调整计划

---

### 2.3 核心区别对比

| 对比项 | RAG问答 | AIOps诊断 |
|--------|---------|-----------|
| **流程** | 一次性 | 多步骤循环 |
| **主动性** | 被动回答 | 主动规划+执行 |
| **工具调用** | 可能调1-2个 | 可能调多个，动态决定 |
| **复杂度** | 简单 | 复杂（需要状态管理） |
| **类比** | 查字典回答 | 医生看病（问诊→检查→诊断） |

## 3. 为什么用LangGraph而不是直接用LangChain

### 3.1 先理解LangChain是什么

**LangChain** = 连接大模型和工具的"胶水"

```python
# LangChain的简单用法
from langchain import OpenAI, PromptTemplate

# 创建链：模板 + 模型
chain = PromptTemplate(template="翻译以下文字：{text}") | OpenAI()

# 调用
result = chain.invoke({"text": "Hello"})
```

**问题**：LangChain的链是**直线型的**，A → B → C，一路到底。

---

### 3.2 LangGraph解决了什么问题

**LangGraph** = 带"记忆"和"分支"的链

#### 生活中的类比

| 场景 | LangChain（简单链） | LangGraph（状态机） |
|------|---------------------|---------------------|
| **做菜** | 按固定步骤：洗菜→切菜→炒菜 | 根据情况调整：如果菜不新鲜→换菜 |
| **看病** | 固定流程：挂号→看病→拿药 | 动态流程：看病→需要检查→等结果→再诊断 |
| **导航** | 固定路线 | 根据实时路况动态调整 |

---

### 3.3 本项目为什么必须用LangGraph

#### AIOps诊断的场景：
```
制定计划 → 执行步骤1 → 看结果 → [结果够了吗？]
                              ↓
                    ┌──────是──────┐
                    ↓              ↓
              生成报告      继续执行步骤2
```

**这里有循环和分支，LangChain做不到！**

#### LangGraph的解决方式：

```python
# 定义状态（记录当前走到哪了）
class PlanExecuteState:
    input: str          # 用户输入
    plan: List[str]     # 待执行计划
    past_steps: List    # 已执行的步骤
    response: str       # 最终答案

# 定义节点（每个节点是一个函数）
def planner(state):    # 制定计划
    ...

def executor(state):   # 执行一步
    ...

def replanner(state):  # 评估是否继续
    ...

# 构建图：节点 + 边
workflow = StateGraph(PlanExecuteState)
workflow.add_node("planner", planner)
workflow.add_node("executor", executor)
workflow.add_node("replanner", replanner)

# 连接：planner → executor → replanner → (判断) → executor 或 结束
workflow.add_edge("planner", "executor")
workflow.add_edge("executor", "replanner")
workflow.add_conditional_edges("replanner", should_continue)
```

---

### 3.4 总结对比

| 特性 | LangChain | LangGraph |
|------|-----------|-----------|
| **结构** | 直线链 | 图（节点+边） |
| **循环** | ❌ 不支持 | ✅ 支持 |
| **分支** | ❌ 不支持 | ✅ 支持 |
| **状态** | 简单传递 | 持久化存储 |
| **适用场景** | 单次调用 | 多轮Agent |
| **本项目** | RAG问答（简单） | AIOps诊断（复杂） |

## 4. Plan-Execute-Replan模式是什么

### 4.1 一句话解释
让AI像人一样"先想后做"，根据情况动态调整计划。

### 4.2 生活中的例子：修电脑

**Plan（制定计划）**：
```
维修师傅说："电脑开不了机，我计划这样排查：
1. 检查电源线
2. 检查电池
3. 检查主板"
```

**Execute（执行）**：
```
师傅做了第1步：检查电源线 → 发现没问题
```

**Replan（重新规划）**：
```
师傅想："电源线没问题，可能不是电源问题，我调整一下计划：
1. 检查电池（提前）
2. 如果电池也没问题，再检查主板"
```

**循环直到**：找到问题或者确定所有可能都检查过了。

---

### 4.3 代码中的实现

#### 文件位置
- `app/agent/aiops/planner.py` - 制定计划
- `app/agent/aiops/executor.py` - 执行步骤
- `app/agent/aiops/replanner.py` - 评估和重规划

#### 状态定义（记录走到哪了）
```python
# app/agent/aiops/state.py
class PlanExecuteState(TypedDict):
    input: str              # 原始任务（如"诊断系统告警"）
    plan: List[str]         # 待执行步骤（如["查日志", "查监控", "分析"]）
    past_steps: List        # 已完成步骤和结果
    response: str           # 最终答案
```

#### 流程图
```
┌─────────┐     ┌──────────┐     ┌──────────┐
│  开始   │────▶│ Planner  │────▶│ Executor │
└─────────┘     └──────────┘     └────┬─────┘
                                      │
                                      ▼
                               ┌──────────┐
                               │ Replanner│
                               └────┬─────┘
                                    │
                     ┌──────────────┼──────────────┐
                     │              │              │
                     ▼              │              ▼
              ┌──────────┐          │        ┌──────────┐
              │ 还有步骤？│───是─────┘        │ 生成报告 │
              └──────────┘                      └──────────┘
                     │
                     否
                     ▼
               ┌──────────┐
               │ 结束     │
               └──────────┘
```

#### 每个节点的职责

**Planner节点**（`planner.py`）：
```python
async def planner(state: PlanExecuteState):
    # 输入：用户任务
    # 输出：执行计划
    
    # 1. 先查知识库，看有没有相关经验
    experience = await search_knowledge(state["input"])
    
    # 2. 让大模型制定计划
    plan = llm.generate(f"根据经验：{experience}，制定计划...")
    
    # 3. 返回计划
    return {"plan": plan}  # 如["查日志", "查监控"]
```

**Executor节点**（`executor.py`）：
```python
async def executor(state: PlanExecuteState):
    # 输入：当前状态（包含plan和past_steps）
    # 输出：执行结果
    
    # 1. 取出第一个待执行步骤
    task = state["plan"][0]
    
    # 2. 执行（调用工具）
    result = await call_tools(task)
    
    # 3. 返回更新：移除已完成步骤，添加到历史
    return {
        "plan": state["plan"][1:],  # 剩余步骤
        "past_steps": [(task, result)]  # 已完成的
    }
```

**Replanner节点**（`replanner.py`）：
```python
async def replanner(state: PlanExecuteState):
    # 输入：当前状态
    # 输出：新的计划 或 最终答案
    
    # 让大模型判断：够不够？还是继续？
    decision = llm.generate(f"已执行：{state['past_steps']}，够了吗？")
    
    if "够了" in decision:
        return {"response": "诊断报告..."}  # 生成最终报告
    else:
        return {"plan": new_plan}  # 继续执行
```

---

### 4.4 为什么这个模式好

| 优势 | 解释 |
|------|------|
| **可解释性** | 能看到AI的"思考过程"（计划是什么） |
| **可控性** | 每一步都知道在做什么 |
| **容错性** | 某步失败可以调整计划 |
| **灵活性** | 根据中间结果动态调整 |

## 5. 消息修剪策略是怎么设计的

### 5.1 为什么要修剪消息

#### 问题的产生
大模型有**上下文窗口限制**（比如最多处理8K、32K tokens）。

假设：
- 每轮对话消耗500 tokens
- 20轮对话后 → 10000 tokens
- **超过限制！** → 报错或截断

#### 后果
- 丢失重要信息
- 无法保持长期记忆
- API调用成本增加

---

### 5.2 本项目的修剪策略

**策略**：保留系统消息 + 最近6条消息（3轮对话）

#### 代码位置
`app/services/rag_agent_service.py:36-73`

#### 代码详解（带注释）
```python
def trim_messages_middleware(state: AgentState):
    """
    修剪消息历史，防止上下文溢出
    
    策略：
    - 保留第一条系统消息（System Message）
    - 保留最近的 6 条消息（3 轮对话）
    - 当消息少于等于 7 条时，不做修剪
    """
    messages = state["messages"]
    
    # 如果消息数量较少，无需修剪
    # 为什么是7？系统消息1 + 最近对话6 = 7
    if len(messages) <= 7:
        return None  # 不用修剪，直接返回
    
    # 提取第一条系统消息（这是给AI的"人设"指令）
    first_msg = messages[0]
    
    # 保留最近的 6 条消息
    # 为什么是6？用户问 + AI答 = 1轮，6条 = 3轮
    recent_messages = messages[-6:]
    
    # 构建新的消息列表：系统消息 + 最近对话
    new_messages = [first_msg] + list(recent_messages)
    
    # 返回修剪后的消息
    return {
        "messages": [
            RemoveMessage(id=REMOVE_ALL_MESSAGES),  # 清除旧的
            *new_messages  # 插入新的
        ]
    }
```

---

### 5.3 可视化理解

```
原始消息（12条）
┌────┬────┬────┬────┬────┬────┬────┬────┬────┬────┬────┬────┐
│ S1 │ U1 │ A1 │ U2 │ A2 │ U3 │ A3 │ U4 │ A4 │ U5 │ A5 │ U6 │
└────┴────┴────┴────┴────┴────┴────┴────┴────┴────┴────┴────┘
 系统  用户 AI  用户 AI  用户 AI  用户 AI  用户 AI  用户
      第1轮     第2轮     第3轮     第4轮     第5轮  （还没回答）

修剪后（7条）
┌────┬────┬────┬────┬────┬────┬────┐
│ S1 │ U4 │ A4 │ U5 │ A5 │ U6 │ A6 │
└────┴────┴────┴────┴────┴────┴────┘
 系统  用户 AI  用户 AI  用户 AI
      第4轮     第5轮     第6轮
      
丢弃的消息（5条）：早期对话（U1, A1, U2, A2, U3, A3）
```

---

### 5.4 设计考量

| 设计决策 | 原因 |
|----------|------|
**保留系统消息** | 给AI的"人设"和指令不能丢 |
| **保留3轮** | 平衡记忆长度和token消耗 |
| **丢弃早期对话** | 通常早期的对话对当前问题影响最小 |
| **6条 = 3轮** | 确保每轮对话完整（用户+AI） |

---

### 5.5 其他修剪策略（扩展知识）

| 策略 | 说明 | 适用场景 |
|------|------|----------|
| **滑动窗口** | 固定保留最近N条 | 本项目用的就是这种 |
| **Token限制** | 按token数而非消息数 | 不同消息长度不同 |
| **摘要压缩** | 早期对话用摘要替代 | 需要保留长期记忆 |
| **选择性保留** | 保留重要消息，丢弃闲聊 | 需要重要性判断 |

## 6. MCP协议解决了什么问题

### 6.1 先理解问题：工具调用的痛点

#### 场景
你的AI Agent需要调用各种工具：
- 查日志（用Python写的服务）
- 查监控（用Go写的服务）
- 发邮件（调用第三方API）
- 操作数据库（直接连MySQL）

#### 没有MCP时的混乱
```python
# 工具1：查日志（Python函数）
def query_logs(start_time, end_time):
    return requests.post("http://log-service:8080/query", ...)

# 工具2：查监控（Go服务，HTTP接口不同）
def query_metrics(metric_name):
    return requests.get(f"http://monitor:9090/api/v1/query?query={metric_name}")

# 工具3：发邮件（第三方SDK）
def send_email(to, subject, content):
    return email_sdk.send(...)

# 问题：
# 1. 每个工具调用方式都不一样
# 2. 参数格式不统一
# 3. 错误处理各不相同
# 4. 增加新工具要改很多代码
```

---

### 6.2 MCP是什么

**MCP = Model Context Protocol（模型上下文协议）**

通俗理解：**USB接口标准**

```
电脑（AI模型）
   │
   │ 各种设备（工具）
   │  ┌─────────────┐
   └──┤ USB鼠标     │
   │  └─────────────┘
   │  ┌─────────────┐
   └──┤ USB键盘     │
   │  └─────────────┘
   │  ┌─────────────┐
   └──┤ U盘         │
      └─────────────┘
      
统一的USB接口 = 统一的MCP协议
```

---

### 6.3 MCP的组成

#### 1. 工具描述（Schema）
每个工具要告诉AI：我能做什么？需要什么参数？

```python
# 工具的定义（标准格式）
{
    "name": "query_logs",
    "description": "查询系统日志",
    "parameters": {
        "type": "object",
        "properties": {
            "service": {"type": "string", "description": "服务名"},
            "level": {"type": "string", "enum": ["ERROR", "WARN", "INFO"]}
        },
        "required": ["service"]
    }
}
```

#### 2. 调用方式（统一接口）
```python
# 不管是哪个工具，都用同样的方式调用
result = await mcp_client.call_tool(
    name="query_logs",  # 工具名
```

## 7. 工具调用失败怎么处理

### 7.1 问题背景

网络请求会失败：
- 网络抖动
- 服务重启
- 请求超时
- 服务端报错

**直接抛异常？** → Agent会崩溃，用户体验差

---

### 7.2 本项目的解决方案

**指数退避重试 + 优雅降级**

#### 代码位置
`app/agent/mcp_client.py:18-74`

#### 代码详解
```python
async def retry_interceptor(request: MCPToolCallRequest, handler, max_retries=3, delay=1.0):
    """
    MCP工具调用重试拦截器
    
    策略：失败就重试，每次等待时间翻倍（指数退避）
    第1次：等1秒
    第2次：等2秒
    第3次：等4秒
    如果都失败，返回错误信息而不是抛异常
    """
    
    last_error = None
    
    # 最多重试3次
    for attempt in range(max_retries):
        try:
            # 尝试调用工具
            result = await handler(request)
            return result  # 成功就返回
            
        except Exception as e:
            last_error = e
            
            # 如果不是最后一次，等待后重试
            if attempt < max_retries - 1:
                wait_time = delay * (2 ** attempt)  # 指数退避！
                # attempt=0: 1 * (2^0) = 1秒
                # attempt=1: 1 * (2^1) = 2秒
                # attempt=2: 1 * (2^2) = 4秒
                
                await asyncio.sleep(wait_time)
    
    # 所有重试都失败
    error_msg = f"工具 {request.name} 在 {max_retries} 次重试后仍然失败: {str(last_error)}"
    
    # 返回错误结果，而不是抛异常
    return CallToolResult(
        content=[TextContent(type="text", text=error_msg)],
        isError=True  # 标记这是错误结果
    )
```

---

### 7.3 时间线可视化

```
第1次调用
  │
  ├─► 失败！
  │
  ├─► 等待 1秒
  │
第2次调用
  │
  ├─► 失败！
  │
  ├─► 等待 2秒
  │
第3次调用
  │
  ├─► 失败！
  │
  ├─► 等待 4秒
  │
第4次调用（最后一次）
  │
  ├─► 成功？→ 返回结果 ✓
  │
  └─► 失败？→ 返回错误信息 ✓

总共等待时间：1 + 2 + 4 = 7秒
```

---

### 7.4 为什么用指数退避

| 策略 | 间隔 | 问题 |
|------|------|------|
| **固定间隔** | 1s, 1s, 1s | 如果服务压力大，每次都撞上高峰 |
| **指数退避** | 1s, 2s, 4s | 逐渐拉长间隔，给服务恢复时间 |

**类比**：
- 固定间隔 = 每秒敲一次门
- 指数退避 = 先等1分钟，再等2分钟，再等4分钟（给主人更多时间）

### 7.5 优雅降级

## 8. 向量检索Top-K=3，如果相关文档不够怎么办

### 8.1 什么是Top-K

**场景**：知识库有1000个文档片段，用户提问。

**Top-K=3** 的意思：只返回最相似的3个文档。

```
知识库（1000个文档片段）
├─ 文档1: "Redis连接池配置..."
├─ 文档2: "MySQL性能优化..."
├─ 文档3: "Nginx负载均衡..."
│  ...（还有997个）
└─ 文档1000: "Linux常用命令..."

用户问："Redis连接超时怎么办？"

相似度计算：
┌─────────────────┬────────────┐
│ 文档            │ 相似度分数  │
├─────────────────┼────────────┤
│ 文档1: Redis... │ 0.95 ⭐    │ ← 最相关
│ 文档5: Redis... │ 0.82 ⭐    │ ← 相关
│ 文档3: Nginx... │ 0.45       │ ← 不太相关
│ 文档2: MySQL... │ 0.30       │
│ ...             │ ...        │
└─────────────────┴────────────┘

Top-K=3 返回：文档1、文档5、文档3
```

---

### 8.2 "不够"的两种情况

#### 情况1：知识库里确实没有相关内容

```
用户问："Python异步编程怎么做？"

知识库内容：
- 全是Redis、MySQL、Nginx运维文档
- 没有Python相关内容

相似度计算结果：
┌─────────────────┬────────────┐
│ 文档            │ 相似度分数  │
├─────────────────┼────────────┤
│ 文档X           │ 0.15       │
│ 文档Y           │ 0.12       │
│ 文档Z           │ 0.08       │
│ ...             │ ...        │
└─────────────────┴────────────┘

所有文档相似度都很低！
```

**本项目的处理方式**：
```python
# 代码位置：app/tools/knowledge_tool.py

async def retrieve_knowledge(query: str):
    # 1. 去Milvus检索
    results = milvus.search(query, top_k=3)
    
    # 2. 检查分数，过滤低质量结果
    # 假设阈值是0.3，低于这个算不相关
    filtered_results = [r for r in results if r.score > 0.3]
    
    # 3. 如果没有相关文档
    if not filtered_results:
        return "未在知识库中找到相关内容。"
    
    # 4. 返回找到的文档
    return format_results(filtered_results)
```

#### 情况2：Milvus返回少于K个

```python
# 假设知识库只有2个文档
results = milvus.search(query, top_k=3)
# 实际返回：2个（因为只有2个文档）

# 代码会正常处理，拿这2个去回答
```

**这种情况代码自动处理**，不会报错。

---

### 8.3 相似度阈值（重要概念）

```
相似度分数范围：0.0 ~ 1.0

1.0 = 完全一样
0.9 = 非常相关
0.7 = 比较相关
0.5 = 有点关系
0.3 = 可能相关（临界值）
0.1 = 基本不相关

设置阈值（如0.3）：
低于0.3的文档 → 认为不相关 → 不返回
```

---

### 8.4 面试回答要点

**Q**: "Top-K=3，如果相关文档不够怎么办？"

**A**:
1. **Milvus会返回实际匹配数量**，比如只有2个相关就返2个，不会硬凑
2. **设置相似度阈值**，过滤掉低质量匹配
3. **处理空结果**，返回友好提示"未找到相关内容"
4. **代码容错**，即使返回0个也能正常处理

## 9. 流式输出SSE是怎么实现的

### 9.1 什么是流式输出

#### 普通输出（一次性返回）
```
用户问：讲个故事

等待...（3秒后）

AI回答：（一次性显示）

```

#### 流式输出（逐字显示）

```
"有"
"座"
"山"
"，"
"山"
"里"
...（像打字机效果）
```

---

### 9.2 SSE是什么

**SSE = Server-Sent Events（服务器发送事件）**

**原理**：HTTP长连接，服务器可以多次推送数据

```
普通HTTP请求：
客户端 ──请求──▶ 服务器
客户端 ◀──响应── 服务器（只有一次）

SSE：
客户端 ──连接──▶ 服务器
客户端 ◀──数据1─ 服务器
客户端 ◀──数据2─ 服务器
客户端 ◀──数据3─ 服务器
...（可以持续推送）
```

**应用场景**：
- ChatGPT的打字机效果
- 股票实时行情
- 直播间弹幕

---

### 9.3 本项目的实现

#### 后端代码（`app/services/rag_agent_service.py:235-304`）
```python
async def query_stream(self, question: str, session_id: str):
    """
    流式处理用户问题
    用 yield 关键字，一次返回一点
    """
    
    # 初始化
    await self._initialize_agent()
    
    # 构造消息
    messages = [SystemMessage(content=...), HumanMessage(content=question)]
    
    # 关键：用 astream 而不是 ainvoke
    # astream = async stream（异步流）
    async for token, metadata in self.agent.astream(
        input={"messages": messages},
        config={"configurable": {"thread_id": session_id}},
        stream_mode="messages",  # 按消息流式输出
    ):
        # 每次拿到一个字/词
        if message_type in ("AIMessage", "AIMessageChunk"):
            content = extract_content(token)
            if content:
                # yield 关键字：每次产生一点数据
                yield {
                    "type": "content",
                    "data": content,  # 这一小块内容
                    "node": node_name
                }
    
    # 最后发个"完成"信号
    yield {"type": "complete"}
```

**关键点**：
- `async for`：异步循环，等数据来
- `yield`：生成器，一次返回一点
- `stream_mode="messages"`：按消息块流式

#### API层（`app/api/chat.py`）
```python
from sse_starlette.sse import EventSourceResponse

@router.post("/chat_stream")
async def chat_stream(request: ChatRequest):
    """
    流式对话接口
    """
    # 调用服务的流式方法
    async def event_generator():
        async for event in rag_agent_service.query_stream(
            question=request.question,
            session_id=request.id
        ):
            # 每个event转成SSE格式
            yield {
                "event": "message",  # SSE事件名
                "data": json.dumps(event)  # JSON数据
            }
    
    # 返回SSE响应
    return EventSourceResponse(event_generator())
```
