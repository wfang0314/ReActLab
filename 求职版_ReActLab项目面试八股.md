## 一、面试前必须统一的项目口径

### 1. 一分钟项目介绍

**面试官问题：请介绍一下 ReActLab 项目。**

**回答要点**

- 定位：可复用的 AI Agent 开发与评测平台，AIOps 是演示场景。
- 两条主链路：RAG 对话 Agent；Plan-Execute-Replan 诊断 Agent。
- 核心技术：FastAPI、LangChain/LangGraph、通义千问、MCP、Milvus、SSE。
- 工程能力：工具调用、会话状态、重试降级、文档索引、Benchmark。
- 边界：MCP 当前是 Mock 数据；测试体系和生产化仍需完善。

**口述的完整回答**

“ReActLab 是一个基于 LangGraph 的 AI Agent 实验和评测平台，项目用智能运维作为演示场景，但核心 Runtime 可以迁移到其他需要多步骤工具调用的任务。系统有两条主要链路：第一条是 RAG 对话 Agent，它通过 Milvus 检索知识文档，并结合本地工具和 MCP 工具回答问题；第二条是 Plan-Execute-Replan 诊断 Agent，Planner 先生成计划，Executor 逐步调用日志、监控或知识库工具，Replanner 根据中间结果决定继续、重规划还是输出报告。后端使用 FastAPI 和 SSE 提供流式接口，模型使用通义千问，MCP 用于解耦工具服务。项目目前能完整跑通，但 MCP 数据主要是模拟数据，自动化测试和生产级可观测性仍是我认为下一步要补强的部分。”

**可能追问**

- 哪一部分是你重点学习或负责的？
- 为什么不直接调用 HTTP API？
- 这个项目离生产环境还有多远？

### 2. 系统请求是怎么走的？

**回答要点**

- Web 前端调用 FastAPI。
- `/api/chat_stream` 进入 RAG Agent；`/api/aiops` 进入 LangGraph 工作流。
- Agent 调用本地工具或从已配置 MCP Server 获取的工具。
- RAG 经 Embedding 检索 Milvus；结果回传模型生成答案。
- 后端通过 SSE 返回事件，前端用 `fetch` 读取响应流。

**口述的完整回答**

“以流式问答为例，前端向 `/api/chat_stream` 发 POST 请求，FastAPI 将问题和 session ID 交给 RAG Agent。Agent 使用相同 session ID 作为 LangGraph 的 thread ID，通过 MemorySaver 读取或更新对话状态。模型判断是否需要调用工具：知识类问题会调用 `retrieve_knowledge`，它把查询向量化后从 Milvus 取 Top-K 文档；运维实时信息则可能调用 MCP 日志或监控工具。工具结果再交给模型组织答案。服务端用 EventSourceResponse 按 SSE 格式持续发送内容块，前端不是原生 EventSource，而是用 fetch 加 `response.body.getReader()` 读取 POST 响应流并增量渲染。”

### 3. 项目最有价值的设计是什么？

**回答要点**

- 将模型推理、流程编排、工具协议、知识检索分层。
- Agent Runtime 与 AIOps 业务场景解耦。
- 中间步骤可观察，失败可以记录并继续。
- 不是单纯“套一个聊天接口”。

**口述的完整回答**

“我认为最有价值的是分层和状态化。LangGraph 负责 Planner、Executor、Replanner 的流程控制，MCP 负责统一外部工具接口，Milvus RAG 负责私有知识，FastAPI 负责交互和流式输出。这样更换模型、增加工具或替换业务场景时，不需要重写整个系统。另外每个执行步骤及结果都保存在状态中，前端可以看到计划和执行过程，出现工具错误时也能把错误作为证据继续交给 Replanner，而不是整个请求直接崩溃。”

- `MemorySaver` 是进程内状态，不适合多实例。
- 消息裁剪函数未接入 Agent。
- RAG 没有相似度阈值、混合检索和 rerank。
- 没有正式 tests 目录。
- Benchmark 中部分结论需要谨慎解释。

**口述的完整回答**

“我不会把这个项目描述成已经生产化。第一，两个 MCP Server 当前主要生成模拟日志和监控数据；第二，LangGraph 使用 MemorySaver，重启和多实例一致性不足，虽然还有 JSON 文件会话存储，但两套状态源没有完全统一；第三，代码定义了消息裁剪函数，但创建 Agent 时没有真正接入；第四，当前 RAG 主要是向量 Top-K，没有阈值过滤、BM25 混合检索和 rerank；第五，仓库没有正式自动化测试目录。它现在更适合作为架构验证和作品集，生产化需要继续补齐测试、状态存储、安全、监控和真实数据源。”

## 二、AI Agent 开发岗位高频题

### A1. 什么是 Agent？它与普通大模型问答有什么区别？

**回答要点**

- LLM 负责生成；Agent 在 LLM 外加入目标、状态、工具和控制循环。
- Agent 能观察结果并决定下一步。
- 风险：不确定性、循环、错误工具调用和成本。

**口述的完整回答**

“普通大模型问答通常是输入 Prompt、生成一次结果；Agent 则是在模型外增加状态、工具和控制循环。模型不仅生成文字，还要根据目标选择工具、读取工具返回的观察结果，再决定继续调用、调整计划还是结束。因此 Agent 更适合跨系统、多步骤任务，但也引入了新的工程问题，比如工具参数错误、无限循环、状态污染、成本不可控以及不可重复。ReActLab 的 AIOps 链路就是用 LangGraph 显式控制这些步骤。”

---

### A2. ReAct 与 Plan-Execute-Replan 有什么区别？

**回答要点**

- ReAct：Reason/Act/Observe 小步循环，灵活但可能短视。
- Plan-Execute-Replan：先给全局计划，再执行和动态修订。
- 本项目普通对话接近工具调用 Agent；诊断链采用后者。

**口述的完整回答**

“ReAct 强调在每一步进行推理、行动和观察，模型根据刚获得的结果立即决定下一步，优点是灵活，缺点是复杂任务中容易局部最优或反复调用。Plan-Execute-Replan 会先建立一个全局计划，Executor 按步骤执行，Replanner 再依据累计证据调整剩余计划，比较适合故障诊断这类有明确阶段的任务。本项目的普通 RAG 对话使用 `create_agent` 做工具调用，而 AIOps 明确采用 Planner、Executor、Replanner 三节点状态图。”

---

### A3. 为什么使用 LangGraph？

**回答要点**

- 图式编排、条件边、循环、持久化状态。
- 每个节点职责明确，便于测试与观测。
- LangChain 并非完全不能做分支；准确说 LangGraph 更适合显式有状态工作流。

**口述的完整回答**

“我选择 LangGraph 的准确理由不是‘LangChain 完全不能做分支’，而是 LangGraph 对有状态、可循环的 Agent 工作流提供了更清晰的抽象。项目把 Planner、Executor、Replanner 定义为节点，用普通边连接规划和执行，用条件边决定继续执行还是结束；状态由 TypedDict 定义，并通过 Checkpointer 按 thread ID 保存。这样流程、状态更新和终止条件都是显式的，比把循环全部写在一个函数里更容易调试、测试和扩展。”

---

### A4. `PlanExecuteState` 是怎么设计的？

**回答要点**

- `input`：原任务。
- `plan`：剩余步骤。
- `past_steps`：步骤和结果；Annotated + operator.add。
- `response`：最终输出。
- 区分覆盖更新与追加更新。

**口述的完整回答**

“状态里有四个核心字段：input 保存原始任务；plan 保存尚未执行的步骤；past_steps 保存已经执行的 `(步骤, 结果)`；response 保存最终报告。关键点是 past_steps 使用 `Annotated[List[tuple], operator.add]`，节点每次只返回本轮新增的一条历史，LangGraph 会自动追加，而不是覆盖之前的证据。plan 则由 Executor 返回 `plan[1:]` 覆盖为剩余计划，Replanner 也可以用新步骤整体替换它。”

---

### A5. Planner 如何保证输出可解析？

**回答要点**

- Pydantic `Plan` 模型，字段为 `steps: List[str]`。
- `with_structured_output` 约束模型输出。
- Prompt 注入工具描述和 RAG 经验。
- 异常时返回默认计划，但默认计划可能无法落到具体工具。

**口述的完整回答**

“Planner 不是让模型自由输出一段 Markdown，而是定义 Pydantic 的 Plan 模型，要求返回 steps 字符串列表，再通过 `with_structured_output(Plan)` 获得结构化结果。规划前会先调用知识检索取得相关经验，同时从本地工具和已配置的 MCP Server 获取工具列表，把工具名称、描述和参数放入 Prompt。异常时会降级为‘收集信息、分析数据、生成报告’三个默认步骤。不过这个降级计划比较抽象，生产上我会增加计划校验，确保每个信息采集步骤都能映射到真实工具。”

---

### A6. Executor 是如何完成一次工具调用的？

**回答要点**

- 取 `plan[0]`。
- 模型绑定全部工具并首次推理。
- 有 tool calls 时交给 ToolNode。
- 将 ToolMessage 加回消息，让模型形成步骤结果。
- 移除当前计划并追加执行历史。

**口述的完整回答**

“Executor 每轮只处理第一个计划步骤。它加载本地和 MCP 工具，用 `bind_tools` 将工具 Schema 绑定到 ChatQwen。第一次调用模型的目标是判断是否需要工具并生成结构化 tool calls；如果有，就由 LangGraph 的 ToolNode 执行，把工具结果作为 ToolMessage 加回上下文，再调用一次模型总结本步骤。如果没有 tool calls，就直接采用模型输出。最后返回剩余 plan 和本轮 past_steps。即使异常，也会移除当前步骤，并把失败原因追加到历史，供 Replanner 判断。”

---

### A7. Replanner 如何防止无限循环？

**回答要点**

- 三种决策：continue、replan、respond。
- `past_steps >= 8` 强制生成响应。
- 执行至少 5 步后禁止 replan。
- 新计划不能多于当前剩余步骤。
- 仍应加入图级 recursion limit、超时和 token/cost budget。

**口述的完整回答**

“Replanner 通过决策约束和硬限制控制循环。模型只能在 continue、replan、respond 中选择；如果累计执行达到 8 步，代码会直接生成最终响应；执行达到 5 步后不再允许重规划；重规划产生的新步骤也不能比当前剩余步骤更多。这能减少计划膨胀，但还不是完整的生产保护。我会再增加 LangGraph recursion limit、整条任务超时、单工具超时、Token 或费用预算，以及重复工具调用检测。”

---

### A8. MCP 是什么？本项目为什么使用它？

**回答要点**

- MCP 标准化模型与工具/资源的交互。
- Schema 可被 Agent 获取并绑定。
- 解耦 Agent 与日志、监控服务。
- 本项目服务器地址仍由配置指定，“发现工具”不等于发现任意服务器。

**口述的完整回答**

“MCP 是模型上下文协议，它把工具名称、描述、输入 Schema 和调用结果标准化。项目将日志查询和监控查询分别做成 FastMCP Server，Agent 通过 MultiServerMCPClient 连接配置好的 8003 和 8004 服务，再调用 `get_tools()` 获取这些服务器暴露的工具。这样 Agent 不需要为每个外部系统写不同的适配逻辑。不过我会准确描述为‘从已配置的 MCP Server 动态获取工具’，而不是说它能自动发现网络上的任意服务。”

---

### A9. MCP 调用失败如何处理？

**回答要点**

- Interceptor 统一处理横切逻辑。
- 最多总尝试 3 次。
- 失败后等待 1 秒、2 秒。
- 最终返回 `isError=True` 的 CallToolResult，而不是抛出。
- 可改进：抖动、可重试错误分类、熔断、幂等性。

**口述的完整回答**

“项目在 MCP 客户端前增加了 retry interceptor。一次工具调用失败后按指数退避等待，默认 `max_retries=3` 的含义是总共最多尝试三次，所以等待间隔是 1 秒和 2 秒，而不是四次调用或等待到 4 秒。全部失败后不继续抛异常，而是返回 `isError=True` 的 CallToolResult，让 Agent 能看到失败证据并继续处理。生产环境还应只重试超时、限流等瞬时错误，增加随机抖动和熔断，并确认写操作是否幂等，避免重复执行。”

---

### A10. RAG 的完整数据链路是什么？

**回答要点**

- 上传只支持 txt/md，最大 10 MB。
- Markdown 标题切分 + RecursiveCharacter 二次切分。
- text-embedding-v4，1024 维。
- Milvus IVF_FLAT + L2。
- 查询 Top-K，格式化来源和标题后作为工具结果。

**口述的完整回答**

“文档通过 `/api/upload` 上传，目前只接受 txt 和 md，单文件最大 10 MB。索引时先按 Markdown 标题保留层级元数据，再用 RecursiveCharacterTextSplitter 对较长内容二次切分；配置的基础 chunk 是 800、overlap 是 100，不过二次分割实际使用了两倍 chunk size。文档用 DashScope 的 text-embedding-v4 转成 1024 维向量，存入 Milvus，索引类型是 IVF_FLAT，距离度量是 L2。查询时 `retrieve_knowledge` 取配置的 Top-K，默认 3，再把文档标题、来源和内容格式化给 Agent。”

---

### A11. Top-K=3 有什么问题？如何优化？

**回答要点**

- 固定 K 不能保证相关性。
- 当前主 RAG 工具没有相似度阈值。
- L2 越小越近，不能当作 0～1 相似度。
- 优化：阈值、混合检索、召回后 rerank、动态 K、引用。

**口述的完整回答**

“Top-K 只保证返回最靠前的 K 条，不保证这些结果真的相关。当前 `retrieve_knowledge` 没有相似度阈值，所以知识库中没有答案时也可能取回弱相关片段。另外项目使用 L2 距离，数值越小越相似，不能把它直接解释成 0 到 1、越大越好的相似度。我的优化顺序是先离线标注数据确定距离阈值，再加入 BM25 与向量的混合召回，把候选扩大后用 reranker 精排，最后根据置信度动态选择 K，并在答案中保留来源引用。”

---

### A12. 会话记忆是怎么实现的？有什么问题？

**回答要点**

- MemorySaver + thread ID。
- JSON 文件 SessionStore 作为前端历史和部分 fallback。
- 双状态源可能不一致。
- MemorySaver 不适合多实例。
- 消息裁剪函数当前未注册。

**口述的完整回答**

“Agent 运行状态主要由 LangGraph MemorySaver 保存，前端传来的 session ID 被用作 thread ID，使同一会话可以接续上下文。项目还把聊天记录保存成 JSON 文件，供历史接口读取和异常时回退。但这形成了内存 Checkpoint 和文件历史两个状态源，存在同步不一致风险；MemorySaver 也无法满足服务重启和多实例共享。还有一个重要细节是代码虽然实现了保留系统消息和最近几轮消息的裁剪函数，但创建 Agent 时没有注册，所以当前实际上未生效。生产上我会统一到 Postgres 或 Redis Checkpointer，并用 Token 数做裁剪或摘要。”

---

### A13. 为什么使用 SSE，而不是 WebSocket？

**回答要点**

- 该场景主要是服务端向客户端单向推送。
- SSE 基于 HTTP，部署和代理更简单，支持事件类型。
- 本项目 POST + fetch reader，非浏览器原生 EventSource。
- WebSocket 适合高频双向交互。

**口述的完整回答**

“聊天生成和诊断进度主要是服务端持续向客户端推送，客户端在一次请求中不需要不断反向发消息，所以 SSE 足够，而且它基于 HTTP，经过网关和代理更简单。后端用 EventSourceResponse 发送 content、plan、step_complete、complete 等事件。因为接口是 POST，前端没有用只支持 GET 的原生 EventSource，而是通过 fetch 获取 Response，再用 ReadableStream reader 解析数据。若要做实时语音、协同编辑或客户端中途频繁控制任务，WebSocket 会更合适。”

---

### A14. 如何降低 Agent 幻觉？

**回答要点**

- Prompt 要求基于真实工具结果。
- 结构化输出减少格式错误，不等于消除事实幻觉。
- 工具结果带来源；无证据时拒答或降级。
- 对最终回答做 claim-evidence 校验。

**口述的完整回答**

“项目已经通过系统提示词要求不得编造，并让诊断尽量基于日志、监控和知识库结果，同时 Planner 和 Replanner 使用结构化输出降低控制字段出错。但结构化输出只能保证格式，不能保证事实。进一步我会让每个结论绑定具体 ToolMessage 或文档来源；检索低置信度时明确回答证据不足；对关键数字和服务名做程序化校验；最终报告再经过一个 evidence checker，检查每条根因是否能追溯到已有证据。”

### A15. 如果把项目扩展到生产环境，你会如何排序？

**回答要点**

- 第一优先级：真实性和安全性。
- 第二：可靠性和状态持久化。
- 第三：可观测性和评测。
- 第四：检索、模型与成本优化。

**口述的完整回答**

“我会先保证正确和安全，而不是先换更大的模型。第一步接入只读的真实日志和监控 API，对工具做鉴权、参数白名单、审计和敏感信息脱敏；第二步增加超时、熔断、幂等、持久化 Checkpointer 和任务恢复；第三步建立节点耗时、工具成功率、Token 成本、检索命中率和端到端任务成功率监控，并补齐回归集；最后再做混合检索、rerank、模型路由和缓存。这个顺序能避免系统能力越强，错误影响反而越大。”

### T1. 你会如何测试这个 Agent 系统？

### T3. 如何测试 MCP 重试？

**回答要点**

**口述的完整回答**

### T4. 如何测试 SSE 流式接口？

**回答要点**

**口述的完整回答**

**回答要点**

## 四、数据分析岗位高频题

### D1. Top-1 Accuracy、Recall@3、MRR 分别是什么？

**回答要点**

- Top-1：第一条是否命中。
- Recall@3：前三条是否至少有一条命中；当前脚本更接近 Hit Rate@3。
- MRR：第一个正确结果排名倒数的均值。
- 指标需结合业务目标。

**口述的完整回答**

“Top-1 Accuracy 表示每个查询排第一的文档是否正确，再对所有查询取平均；Recall@3 看正确文档是否出现在前三。严格来说，如果每个问题只定义一个目标文档，这个实现更接近 Hit Rate@3；真正 Recall@3 通常要除以该查询全部相关文档数。MRR 是第一个正确结果排名的倒数均值，例如排第一得 1，排第二得 0.5。Top-1 反映直接可用性，MRR 对排序位置更敏感，前三命中则更适合后续有 rerank 或多文档生成的场景。”

---

### D2. 如何解读仓库中的 86%、94% 和 0.8867？

**回答要点**

- 在存量 JSON 中是 100 条查询上的 Top-1、前三命中和 MRR。
- 不能直接外推到全部用户问题。
- 查询均匀分布于 5 个故障类别，且英文模板化明显。
- 需要置信区间、分层指标和独立测试集。

**口述的完整回答**

“仓库记录的结果表示在那一版 100 条测试查询中，86 条第一名来源正确，94 条在前三名中命中，首个正确结果的平均倒数排名是 0.8867。这说明在该数据集上排序表现较好，但不能直接说线上准确率就是 86%。数据主要覆盖 CPU、内存、磁盘、慢响应和服务不可用五类，而且每类数量相近、问题模板比较集中。更严谨的做法是按故障类型、表达语言和难度分层，给出 bootstrap 置信区间，并使用与开发集隔离的测试集。”

---

### D3. 如何构建更可靠的 RAG 评测数据集？

**回答要点**

- 从真实问题采样，去重和脱敏。
- 多相关文档标注，不只一个文件名。
- 正例、困难负例、知识库外问题。
- 双人标注和一致性检查。
- 按意图、难度、语言、时间切分。

**口述的完整回答**

“我会从脱敏后的真实搜索词、工单和客服问题中分层采样，而不是只人工改写模板。每个查询由至少两名标注者标注相关文档及相关等级，因为一个问题可能对应多篇有效资料；分歧通过复核解决，并计算一致性。数据应包含简单正例、术语歧义、跨文档问题、困难负例和知识库外问题。切分时按来源或时间隔离，避免同一模板的改写同时出现在训练和测试集，造成数据泄漏。”

---

### D4. 如何比较 Top-K=3 和 Top-K=5？

**回答要点**

- 明确假设：K 增大提升召回，但可能降低上下文精度、增加成本。
- 同一查询集配对比较。
- 同时看 Recall、nDCG、生成正确率、Token 和延迟。
- 对每个 query 的差值做 bootstrap CI。

**口述的完整回答**

“我会做配对实验，保持 Embedding、索引、文档切分、模型和 Prompt 都不变，只改变 K。离线看 Recall@K、MRR 或 nDCG；端到端还要看答案正确率、忠实度、引用准确率、输入 Token、延迟和费用。因为同一查询在两个方案下都有结果，可以对逐查询差值做 bootstrap 置信区间。若 K=5 只提升少量召回，却带来更多噪声和明显成本，就不应只因为 Recall 更高而上线。”

---

### D5. 什么是 A/B 测试？如何评估重试机制？

**回答要点**

- 对照组无重试，实验组有重试。
- 保持请求分布和故障注入一致。
- 成功率、额外延迟、调用次数、服务负载。
- 按错误类型分层。
- 当前脚本是确定性模拟，不是线上随机 A/B。

**口述的完整回答**

“重试 A/B 的核心不是只比较成功率。对照组禁用重试，实验组采用相同的最大次数和退避策略，向两组注入相同分布的超时、限流和永久错误。比较最终成功率、P95 延迟、平均调用次数、下游负载以及重复副作用率，并按错误类型分层。仓库现有测试用 Mock Handler 设定失败次数，属于确定性故障注入或机制验证，不是严格的线上随机 A/B，因此我会称它为重试策略模拟实验。”

### D6. 如何分析 Agent 失败案例？

**回答要点**

- 建立失败分类：理解、规划、工具选择、参数、工具、证据整合、终止。
- 保存 trace，定位首次偏离节点。
- 统计频率、影响、可修复性。
- 混淆矩阵或 Sankey 可看错误流转。

**口述的完整回答**

“我会先定义失败 taxonomy：意图理解错误、计划缺步骤、工具选择错误、参数错误、工具服务故障、检索失败、证据总结错误、过早结束和循环超限。对每个任务保存节点状态、tool call、tool result、耗时和 Token，标注首次发生偏离的节点，而不是只标最终失败。然后按频率、用户影响和修复成本排序。例如大量失败来自参数格式，就优先优化 Schema 和校验；如果来自证据整合，再考虑 Prompt 或模型。”

### D7. 如何建立 Agent 线上指标体系？

**回答要点**

- 北极星：有效任务完成率或用户问题解决率。
- 质量：证据支持率、工具选择准确率、检索指标。
- 效率：TTFT、总延迟、步骤数、Token、费用。
- 可靠性：工具错误、超时、循环、降级。
- 产品：采纳率、重问率、人工转接率。

**口述的完整回答**

“我会以有效任务完成率作为核心指标，而不是只看调用量。质量层监控有证据支持的结论比例、工具选择和参数正确率、检索命中及用户评分；效率层看首 Token、总延迟、平均步骤、Token 和单任务费用；可靠性层看工具错误率、重试恢复率、超时率和循环终止率；产品层看建议采纳率、同问题重问率和人工接管率。所有指标按任务类型、模型版本、工具和用户群体切片，避免总平均掩盖局部退化。”

---

### D8. 如何判断指标提升是否显著？

**回答要点**

- 不只看百分点。
- 二元成功率可用比例检验或配对检验。
- 排序指标和复杂分布用 bootstrap。
- 报置信区间、效应量、多重检验问题。
- 统计显著不等于业务显著。

**口述的完整回答**

“我会同时报告点估计、置信区间和效应量。对同一批查询的新旧方案，Top-1 命中是配对二元结果，可以使用 McNemar 检验；MRR、延迟等不一定正态，可对逐查询差值做 bootstrap。若同时比较很多 K、Prompt 和模型，需要控制多重检验或预先定义主指标。最后还要判断业务意义，例如命中率提升 0.5 个百分点即使统计显著，若成本增加 30%，也未必值得上线。”

---

### D9. 为什么平均延迟不够？

**回答要点**

- 延迟通常右偏，长尾影响用户。
- P50 表典型，P95/P99 表尾部。
- 流式系统要看 TTFT 和总完成时间。
- 当前 curl 子进程法有测量偏差。

**口述的完整回答**

“延迟通常是右偏分布，少量慢请求会严重影响用户，平均值既可能被极端值拉高，也看不出尾部 SLA。因此要至少报告 P50、P95、P99。Agent 流式接口还应拆成首 Token 时间和完整任务时间，因为用户体感更受首 Token 影响。当前 Benchmark 用启动 curl 子进程测健康接口 P50，会包含客户端启动开销；更严谨的方式是复用连接的压测客户端，并分别记录客户端和服务端埋点。”

---

### D10. 如果向业务方做仪表盘，你会展示什么？

**回答要点**

- 总览：任务数、完成率、满意度、成本。
- 漏斗：请求→规划→工具成功→报告→采纳。
- 趋势和分层。
- 失败 Top 原因和案例钻取。
- 避免只有技术指标。

**口述的完整回答**

“首页会展示任务量、有效完成率、用户采纳或满意度、P95 延迟和总成本。第二层做执行漏斗：收到请求、成功规划、工具调用成功、生成报告、用户采纳，快速定位损失发生在哪一步。再按任务类型、模型、工具、时间和服务切片趋势，并展示失败原因 Top N。每个异常点可以下钻到脱敏 trace。这样业务方能看到系统是否真正解决问题，研发也能定位具体技术瓶颈。”

## 五、跨岗位系统设计追问

### C1. 为什么选择 Milvus？

**回答要点**

- 面向向量检索，支持索引、元数据和规模扩展。
- 项目本地用 standalone，部署包含 etcd、MinIO、Attu。
- 小规模项目也可用更轻量方案；选型看规模和运维成本。

**口述的完整回答**

“Milvus 专注向量检索，支持多种索引、向量字段和元数据，适合后续扩展较大知识库。项目用 standalone 模式，etcd 管元数据，MinIO 存对象，Attu 用于可视化管理。不过当前只有少量运维文档，Milvus 并非唯一选择，FAISS、Chroma 或带 pgvector 的 Postgres 也可能更轻。我的选型原则会结合数据规模、过滤需求、一致性、团队已有基础设施和运维成本，而不是因为它流行。”

### C2. IVF_FLAT 和 nprobe 是什么？

**回答要点**

- IVF 先把向量分桶；查询只搜部分桶。
- FLAT 表示桶内精确扫描。
- nprobe 是查询访问的桶数。
- nprobe 越大召回通常越高，但延迟增加。

**口述的完整回答**

“IVF_FLAT 建索引时先用聚类把向量划分到多个倒排桶，查询时先找到接近的桶，再在桶内做精确距离计算。项目搜索参数的 nprobe 是 10，表示一次查询探测若干候选桶。nprobe 提高通常能增加召回，但也会扫描更多向量、增加延迟，所以应在真实查询集上画召回与延迟的权衡曲线，而不是固定一个经验值。”

---

### C3. L2 距离与余弦相似度有什么区别？

**回答要点**

- L2 看绝对欧氏距离，越小越近。
- cosine 看方向，越大越相似。
- 向量归一化后排序关系可能接近。
- 必须和 Embedding 模型建议及索引配置一致。

**口述的完整回答**

“L2 计算两个向量之间的欧氏距离，结果越小越相似；余弦相似度比较方向，结果越大越相似，对向量长度不敏感。如果向量做了单位归一化，两者的排序可能等价或接近，但不能默认如此。项目选择 L2，因此分析 score 时必须按距离解释。我会参考 Embedding 模型推荐并用离线集比较不同度量，确保建索引和查询时使用一致的 metric。”

---

### C4. 如何保证上传索引的幂等性？

**回答要点**

- 文件名规范化并覆盖本地文件。
- 索引前按 source 删除旧数据，再写新分片。
- 风险：删除成功写入失败造成空窗。
- 优化：版本化、事务式切换、内容哈希。

**口述的完整回答**

“当前实现会规范化文件名，重复上传时覆盖 uploads 中的文件；索引前根据规范化 source 删除该文件原有向量，再写入新的分片，因此目标是实现覆盖更新而非重复追加。但删除和新增不是原子操作，如果 Embedding 或写入失败，会出现旧索引已删、新索引未完成的空窗。生产上我会使用内容哈希去重，把新版本写入临时版本，全部成功后再切换 active version，最后异步清理旧数据。”

### C5. 配置和密钥如何管理？

**回答要点**

- Pydantic Settings 从 `.env` 加载。
- `.env.example` 只放键和示例。
- 生产使用 Secret Manager/K8s Secret，不入库、不打印。
- 配置校验应在启动时 fail fast。

**口述的完整回答**

“项目用 Pydantic Settings 从 `.env` 加载模型、Milvus、RAG 和 MCP 配置，类型转换比较清晰。开发环境可以用 `.env`，但真实 Key 不能提交仓库，日志也只能打印脱敏结果；生产环境应使用云 Secret Manager 或 Kubernetes Secret 注入。启动时还要校验必填配置和 URL，关键依赖缺失时快速失败，而不是等到第一个用户请求才暴露问题。”

## 六、简历和面试中的真实性边界

### 可以这样说

- “集成 MCP 日志和监控工具，当前使用 Mock 数据验证调用链路。”

### 不要这样说

- “Agent 十个真实业务任务成功率 100%。”现有脚本只是结构校验。

- “RAG 有 0.3 相似度阈值。”当前主检索工具没有阈值。
- “消息裁剪已经生效。”函数存在，但没有注册到 Agent。
- “重试 3 次意味着首次调用加 3 次重试。”当前总尝试数是 3。
- “系统已经具备生产级高可用。”目前是单机和进程内 Checkpointer。
